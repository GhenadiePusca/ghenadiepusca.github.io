Functions in Swift

April 21, 2020 16 minutes

Swift is a multi-paradigm language, in light terms meaning you can do both Object Oriented Programming and Functional Programming. Even though it can do both, and even at the same time, in same program, we usually focus most of our attention and care to Object Oriented part, and ignoring, leaving or take it as granted what functions can offer. And functions have to offer a lot. In this post we’ll take a high level look what functions can do, beyond their normal usage. This post will also act as a base for the future posts when we will dive deeper in FP(I’ll refer further to functional programming as FP) world with Structure and Interpretation of Computer Programms book.

What do we know

You’ve most probably heard about FP, and almost for sure it had to do something with map, flatMap, reduce, filter and other buzzwords. Maybe someone even said that those are higher order functions. And… whatever, without looking into more details we use those functions without putting much thought into them. Maybe because higher order functions sound scary. More, it is very possible that a lot of us think that map, flatMap are Array specific functionality. Well I sure did, before getting to know what is map in swift, I did now the Map from java, which is a dictionary. But actually map and flatMap have a more deeper implication(we’ll get into this in future posts). Aaand, thats it, we’ve stoped at those functions used to process arrays. Let’s not leave it that way.

What is actually a function?

Looking over the internet you may find a lot of definitions of what is a function. But the idea is the same - it is a block of code that encapsulates a given, reusable functionality. We can refer to an object from OOP, which also encapsulates a given/reusable functionality. We’ll see that the difference between an object and a function is actually very small, and throughout the post I will compare a function to an object/class.

In swift a function usually is of form:

func <name>(args..) -><return type>

Comparing to OOP:

  • name indentifies the function in same way the name identifies a class
  • args are the inputs of the function, in same way class has inputs through the initialiser. However class can define inputs also through its public properties and/or functions.
  • return type is the output of a function, in the same way a class defines public properties and/or method to expose its outputs.

Function depedencies

There is often a need to use the functionality of another function in the current function. As with classes there are two ways of doing so:

  • Implicit depedency - we do call another function directly inside the current function, in the same way we would with classes - directly refer to another class: 
       func <x>() -> Int {
       ...
   }

   func <y>(args...) -> Int {
       let someValue = <x>()  /* <--- implicit depedency */
       ...
   }
  • Explicit dependency - we do inject another function as one of the arguments of the current function, in the same way we would inject an object into another, through initializer for example: 
       func <x>() -> Int {
       ...
   }

   func <y>(f: () -> Int) -> Int {  /* <--- explicit depedency */
       let someValue = f()
       ...
   }

Implicit depedency is very similar between functions and objects, however, explicit dependency in function has one crucial difference:

It forces polymorphism

Let’s compare with OOP with an example.

We can do it this way - requiring in intialiser an instance of a concrete class

class X {
    func process() -> Int {...}
}

class Y {
    init(x: X) {
        ...
    }
}

Or, require a instance of a class which conforms to a protocol:

protocol P {
    func process() -> Int
}

class X: P {
    func process() -> Int {...}
}

class Y {
    init(p: P) {
        ...
    }
}

It is well known that with second approach we enable polymorphic behaviour of a class Y - we can inject different behaviour with a different object of a class which conforms to protocol P.

Now with functions, there is no way you can require to have as an argument a concrete function:

func Y() {...}

func X(y: Y) { ... } /* <--- you can't do this */

Instead functions do force you to define a blueprint of a function through a method signature.

func X(f: () -> Int) { ... }

Where:

 () -> Int

Is the blueprint of a function that function X requires - a function which does not take any parameter and returns an Int. If we look closely we will see that:

protocol P {
    func process() -> Int
}

Is the same as:

() -> Int

Both do define a contract, a requirement. And it is correct to say that:

A protocol with one function can be substituted with just a function, and vice-versa.

Let’s see a complete example:

protocol P {
    func process() -> Int
}

class X: P {
    func process() -> Int {...}
}

class Z: P {
    func process() -> Int {...}
}

class Y {
    init(p: P) {
        ...
    }
}

let y1 = Y(X())
let y2 = Y(Z())

VS:


func process() -> Int {
    ...
}

class Z {
    func compute() -> Int {...}
}

class Y {
    init(f: () -> Int) {
        ...
    }
}

func fY(f: () -> Int) {
    ...
}

let z = Z()
let y1 = Y(process)
let y2 = Y(z.compute)
let fy1 = fY(process)
let fy2 = fY(Z.compute)

As we can see both approaces are very similar. But the key difference is that function dependencies do force you by default to use function blueprints - same as protocols for classes. Also it forces you to follow Interface Segregation principle, even though there are no interfaces in FP, but the ideea is that if for one function you require as an input two other functions, you will most probably do:

func z(f1: () -> Int, f2: (Int) -> Double) -> Double {
    ...
}

while in OOP, without much thought(even though we should) we will do:

protocol P {
    func f1() -> Int
    func f2(i: Int) -> Double
}

class Z {
    init(p: P)
}

We will keep adding functions to protocol P, thus violating Interface Segregation principle. For sure there are cases when two functions make sense to be in the same protocol, but usually we do add more functions to protocol just for convenience.

Function arguments and theyre evaluation

There are different terms in FP to describe how function arguments are evaluated. Two of which I am aware of are:

  • Call by name vs call by value - as per Martin Odersky in Scala
  • Normal order vs applicative order - as per Structure and Interpretation of Compute Programms book

Both are different name for the same things.

Call by value / applicative order

In this case, argumets of a function are first evaluated and then function is applied.

func f(a: Int) -> Int {
    return 5 + a
}

f(a: 9-5)
/// this becomes
f(a: 4)
/// and then
5+4

In example above, first the value of 9-5 will be computed, and then function f will be actually applied with the value of 4. Thus we can also say that:

With call by value / applicative order, arguments of a function are evaluated before those are used.

Call by name / normal order

In this case, function is first applied/expanded and then the value of the arguments are evaluated. As in Scala, in Swift we can achieve this by having as an input a function:

func f(f: () -> Int) -> Int {
    return 5 + f()
}

f(f: { 9-5 })
/// this becomes
5+f()
/// and then
5+4

With call by name / normal order, arguments of a function are not evaluated until those are needed. We can say that it is lazy evaluation.

So what are the benefits of one over another?

With call by value / applicative order we can make the result of the argument reusable, thus improving the program performance:

func f(a: Int) -> Int {
    let x = 10 + a
    let y = 10 - a
    return (x + y)*a
}
f(a: 9 - 5)
/// becomes
let x = 10 + 4
let y = 10 - 4
(x + y)*4

// vs

func fN(f: () -> Int) -> Int {
    let x = 10 + f()
    let y = 10 - f()
    return (x + y)*f()
}

fN(f: { 9 - 5 })
// becomes
let x = 10 + f()
let y = 10 - f()
(x + y)*f()

Note that in the case of call by name / applicative order the function f() will called three times, thus the value 9-5 will be computed 3 times. Sure you can say that we can store the value of f() in variable, but then why not having the function result passed as an argument.

With call by name / normal order we can avoid doing some calculations if not needed:

func f(x: Int, a: Int) -> Int {
    if x < 0 {
        return x
    }
    return x + a
}

f(x: -4, a: 9-5)

/// vs
func fN(x: Int, f: () -> Int) -> Int {
    if x < 0 {
        return x
    }
    return x + f()
}

f(x: -4, a: { 9-5 })

We can see that with call by name / normal order we will calculate the value only when it is needed, while with call by value / applicative order we will calculate the value of the argument a even if it will not be needed. Ofcourse the example above is oversimplified, and most of the times we would use call by value / applicative order. But it also can be a greate improvement for your program to identify the cases when an call by name / normal order would be better. Usually you would do this to avoid allocation of some resources until those are needed. An example would be when you do some request and on error to show an error banner, you will instantiate the error banner on errorCompletion(or something like this) instead of passing an instance directly to the calling function.

Function as return type of another function

Functions can return another function as a result. In this case it will act more as a builder - the primary function being an instantiator of the function to be returned:

func makePrinter(withPrefix: String) -> (String) -> Void {
    return { append in
         print(prefix + append)
    }
} 

// or

func makePrinter(withPrefix: String) -> (String) -> Void {
    /// Also known in swift as nested function
    func show(_ append: String) {
         print(prefix + append)
    }

    return show
} 

let printer = makePrinter(withPrefix: "Hello")
printer("World")
/// prints - Hello World
printer("Coders")
/// prints - Hello coders

To make a reference to OOP, the code above can be translated as:

class Printer {
    private let prefix: String
    init(withPrefix prefix: String) {
        self.prefix = prefix
    }

    func show(_ append: String) {
        print(prefix + append)
    }
}

let printer = Printer(withPrefix: "Hello")
printer.show("World")
/// prints - Hello World
printer.show("Coders")
/// prints - Hello coders

Closures

In previous examples we encountered two types of closures:

  • closure expression
  f(x: -4, a: { 9-5 })
  /// Here { 9 - 5 } is a closure expressions
  • nested function
func makePrinter(withPrefix: String) -> (String) -> Void {
    /// Nested function
    func show(_ append: String) {
         print(prefix + append)
    }

    return show
}

If you are a bit confused why nested function is a closure, lets refer to the apple documentation:

Closures are self-contained blocks of functionality that can be passed around and used in your code.

Thats it - nested functions(and global functions) are special cases of closures. Thus all nested/global functions do have the next important functionality:

Closures can capture and store references to any constants and variables from the context in which they are defined.

We’ll make use of this behaviour in the next section.

Most of us used closure expressions. And we’ve used it so much, that we started to refer to it just as closure(even though it is not 100% correct). Most of its usages is related to completion handlers, and again, by being so common, we’ve started to call the completion handler itself a closure, while it is just a function signature - thus both of the below implementations are correct:

func apiCall(params: ..., completionHandler: (Result<Data, Error>) -> Void) {
    ...
}

apiCall(params..., completionHandler: { result in

})

/// and

apiCall(params..., completionHandler: handleCompletion)

func handleCompletion(_ result: Result<Data, Error>) {
    ...
}

Both completion handlers are exactly the same, main difference being that, one - closure expression - is so called unnamed function while handleCompletion is nammed function. Thus in this case closure expression is just used as a convenience to create a function in place.

Another example that makes heavy use of closure expressions is lazy var

   class A {
       lazy var cl: Int = {
           9 - 5
       }()

       lazy var f: Int = initF()

       private function initF() -> Int {
           9 - 5
       }
   }

Again both are the same, and closure is used as unnamed function that can be created in place without the need of creating a new named function. Though having named function will generally be better for code readability and maintance.

So, from closures we do mostly use closure expression, and even more, most of casses are just for convenience of creating an unnamed function. But we can do with closures, much, much more. Let’s see below.

Class vs function

We saw with printer example that function was able to do the same thing as a class. Can we take it further and say that we don’t need a class to create an object? Let’s take a famous example of banking account, and see if there is a functional difference between objects created with a classs and an object created with functions.

class BankAccount {
    private var balance: Int
    
    init(initialBalance: Int) {
        balance = initialBalance
    }
    
    func getBalance() -> Int {
        balance
    }
    
    func deposit(_ amount: Int) {
        balance += amount
    }
    
    func widthdrawl(_ amount: Int) {
        guard amount < balance else {
            return
        }
        
        balance -= amount
    }
}

let ba = BankAccount(initialBalance: 10)
ba.deposit(5)
ba.widthdrawl(12)
let balance = ba.getBalance() /// outputs 3

A simple implementation of a bank account with a class, can we completely get rid of the class and have only a function?

Let’s do it step by step. First we have to identify what is actually the functionality of a bank account.

From the class we see that there are 3 functions - getBalance, deposit and widthdrawl. It’s fair to say that we can describe a bank account with those 3 functions.

We can group those 3 functions in a tuple and define the bank account object:

typealias FBankAccount = (getBalance: () -> Int,
                          deposit: (Int) -> Void,
                          widthdrawl: (Int) -> Void)

What about the initialiser from the class? We saw previously that we can have functions as builders, we will need something like this:

func makeBanckAcount(withInitialBalance balance: Int) -> FBankAccount {
    ...
}

Then:

func makeBankAcount(withInitialBalance balance: Int) -> FBankAccount {
    func getBalance() -> Int {
        balance
    }
    
    func deposit(_ amount: Int) {
        ???
    }
    
    func widthdrawl(_ amount: Int) {
        ???
    }
    return (getBalance, deposit, widthdrawl)
}

Note the ???. In deposit and widthdrawl it is needed to modify the balance. Most of the times we create objects from classes for those to encapsulate some state, if there is no state involvevd, then we most probably will declare functions as static, and the class will just as an name space.

At least from my perspective objects are means to encapsulate a state and allow operations on that state. If we can describe state in a function then we can say that we did describe an object with functions without the need for a class.

So how can we have state inside a function? As we previosuly saw, closures can capture values. Let’s complete the info we do have with some more form the Apple docs:

A closure can capture any of its outer function’s arguments, variables, constants. The closure can then refer to and modify the values of those constants and variables from within its body, even if the original scope that defined the constants and variables no longer exists.

With banking account, we do have 3 nested functions(which, lets not forget, are closures). And once makeBankAccount returns, those 3 functions will leave the makeBankAccount scope, but the state will be captured.

func makeBankAcount(withInitialBalance balance: Int) -> FBankAccount {
    /// Private state
    var currentBalance = balance
    
    func getBalance() -> Int {
        currentBalance
    }
    
    func deposit(_ amount: Int) {
        currentBalance += amount
    }
    
    func widthdrawl(_ amount: Int) {
        guard amount < currentBalance else {
            return
        }
        
        currentBalance -= amount
    }
    return (getBalance, deposit, widthdrawl)
}

let fBankAccount = makeBankAcount(withInitialBalance: 10)
fBankAccount.deposit(5)
fBankAccount.widthdrawl(12)
let balance = fBankAccount.getBalance() /// outputs 3

Note that currentBalance is private implicitly and it is visible only inside the makeBankAcount scope and 3 nested functions. And I think now we can safely say that fBankAccount is in the same manner an object as the bank account created with a class.

But, you my ask, what about subclassing?

Suppose we do want to have a bank account with interest applied on deposit. In OOP we’ll do something like this:

class SavingsBankAccount: BankAccount {
    private let interestRate: Double
    
    init(interestRate: Double, initialBalance: Double) {
        self.interestRate = interestRate
        super.init(initialBalance: initialBalance)
    }

    override func deposit(_ amount: Double) {
        super.deposit(amount * interestRate / 100)
    }
}

Now with only functions:

func makeSavingsBankAccount(interestRate: Double, initialBalance: Double) -> FSavingsBankAccount {
    let bankAccount = makeBankAcount(withInitialBalance: initialBalance)
    func deposit(_ amount: Double) {
        let interest = amount * interestRate / 100
        bankAccount.deposit(amount + interest)
    }
    return (bankAccount.getBalance, deposit, bankAccount.widthdrawl)
}

You may argue that with functions it is not actual inheritance, and you will be right - Functions cannot inherit. But we have achieved the same behaviour as with inheritance through means of composition.

Conclusion

This was a rather long blog post, but I hope you enjoyed it and maybe learned something new. This post is meant to serve as a base for diving deeper into the functional proramming. We’ve touched some points, but mostly high level, we’ll get into details step by step.

  • We saw that functions are very similar to classes, and that we can represent and object only through functions.
  • We saw the power of closure, and how underused it is, but as always with great power comes great responsibility.
  • We saw that functions have a lot to give, and since most of programs functionality is embeded in functions, leveraging the functionality of functions can be a great tool in your arsenal.

By no means, I do advocate that FP is better than OOP and viceversa. Both are powerful, and I think we can use the best of two worlds to achieve more. Somehow we are getting a bit scared, confused and other funny stuff, when we do hear about functional programming. I guess it is the same when FP guys do hear about OOP. As we saw in this post the line between functions and objects is very thin, so don’t be afraid to use some of the FP goodness in your OOP projects, let it be another tool you can use to build great things.
